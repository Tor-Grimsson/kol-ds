# kol-brand — manifest schema, template, and the Kolkrabbi brand package

Agreed direction 2026-07-03. A **brand kit** is a data+assets package — no
server (unlike the clients tier). The pain it kills: per-client brand data
(ramps, type specs, logos, press/presence info, stationery assets) is copied
from client to client and hand-swapped, dragging stale leftovers along.

## The keystone

> **The schema is the actual product. The template conforms to it, the scraper
> feeds it, stationery consumes it, the styleguide renders it.**

One **brand manifest schema**: a standard ESM shape (`{ meta, colors, ramps,
type, logos, clearspace, stationery, presence, press, timeline, … }`).
Everything else is a view of it. The styleguide machinery (TypeSample / Swatch
/ SpectrumGrid — the E3 pieces) becomes a set of generic renderers of any
conforming package: **new client = new data package, zero new UI.**

**Every field optional** — "fill or leave empty" is a requirement; renderers
must degrade gracefully on missing fields (gaps, not crashes). Build order is
forced: **schema → template → (scraper adapter) → stationery.** Nothing
before the schema.

## Packages

| Package | What | Registry |
|---|---|---|
| `@kolkrabbi/kol-brand-template` | schema (JSDoc-typed) + reference implementation with obviously-dummy placeholder data — the **slate** copied for each new client AND the **dev fixture** the generic styleguide renderers develop against | public |
| `@kolkrabbi/kol-brand` | Kolkrabbi's own real brand data, conforming | public |
| per-client instances | copy of the template, filled by hand and/or seeded by the [[2026-07-03-kol-scrape|scraper]] adapter | **NEVER public npm** — lives in the client's repo as a local package conforming to the schema. Confidentiality rule, non-negotiable. |

Schema lives **inside** `kol-brand-template` initially; extract to its own tiny
package only if a consumer ever needs schema-without-dummy-data (YAGNI).

`kol-theme` stays untouched — it's how the *system* looks (tokens/CSS), not
what a brand *is*.

## Tools are not contents

Scrapers, setup pages, stationery generators are producers/consumers of the
manifest — never inside a data package. Stationery generators become their own
tool package(s) when next needed (no `brand-tools` grab-bag — the clients-tier
lock-step argument applies).
